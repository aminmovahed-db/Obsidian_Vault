The only shared visibility is `pilot_uat`, where all teams get `USE CATALOG` (can see schema names but not other teams' data).

#### Catalog Structure

```
pilot_team_a/
  ├── project_x/                ← schema (one per project)
  │     ├── tables, views, models   (team-created)
  │     └── project_x_files         (volume — DnA-provisioned)
  ├── project_y/
  │     ├── tables, views, models
  │     └── project_y_files         (volume — DnA-provisioned)
  └── default/

pilot_team_b/
  ├── project_z/
  │     ├── tables, views, models
  │     └── project_z_files         (volume — DnA-provisioned)
  └── default/

pilot_uat/
  ├── team_a/                   ← UAT promotion target
  └── team_b/

prod/                           ← read-only for all teams
uat/                            ← read-only for all teams
```

#### 1 — Catalog-Level Grants

| Principal | `pilot_team_a` | `pilot_team_b` | `pilot_uat` | `prod` | `uat` |
|---|---|---|---|---|---|
| `grp_team_a` | `USE CATALOG` | ❌ _no grant_ | `USE CATALOG` | `USE CATALOG` | `USE CATALOG` |
| `grp_team_b` | ❌ _no grant_ | `USE CATALOG` | `USE CATALOG` | `USE CATALOG` | `USE CATALOG` |
| `dna_admin` | `ALL PRIVILEGES` | `ALL PRIVILEGES` | `ALL PRIVILEGES` | `ALL PRIVILEGES` | `ALL PRIVILEGES` |
| `security_job_sp` | `ALL PRIVILEGES` | `ALL PRIVILEGES` | `ALL PRIVILEGES` | — | — |

> **Key point:** Team A cannot even _see_ `pilot_team_b` because they lack `USE CATALOG`. Catalog-level isolation is stronger than schema-level — no schema names, no metadata leak.

#### 2 — Schema-Level Grants

| Principal | `pilot_team_a.project_x` | `pilot_team_a.project_y` | `pilot_team_b.project_z` |
|---|---|---|---|
| `grp_team_a` | `USE SCHEMA`, `CREATE TABLE`, `CREATE VIEW`, `CREATE FUNCTION`, `CREATE MODEL` | `USE SCHEMA`, `CREATE TABLE`, `CREATE VIEW`, `CREATE FUNCTION`, `CREATE MODEL` | ❌ _no grant_ |
| `grp_team_b` | ❌ _no grant_ | ❌ _no grant_ | `USE SCHEMA`, `CREATE TABLE`, `CREATE VIEW`, `CREATE FUNCTION`, `CREATE MODEL` |
| `dna_admin` | `ALL PRIVILEGES` | `ALL PRIVILEGES` | `ALL PRIVILEGES` |

> **Note:** `CREATE VOLUME` is **not** granted — only DnA can provision volumes. `CREATE SCHEMA` is also withheld (DnA creates schemas via Terraform).

#### 3 — Object-Level Grants

| Object type | Grant to team group | Notes |
|---|---|---|
| Tables / Views | `SELECT`, `MODIFY` on _own schema_ | Team members can read/write all tables within their project schemas |
| Models | `EXECUTE`, `SELECT` on _own schema_ | Teams train and serve models freely within their boundary |
| Volumes | `READ VOLUME`, `WRITE VOLUME` | **No** `CREATE VOLUME`, **no** `MANAGE` — DnA creates and owns volumes |
| Functions | `EXECUTE` on _own schema_ | UDFs created by team are available to all team members |
| `prod` / `uat` objects | `SELECT` only | Read-only across all catalogs outside pilot |

#### 4 — `pilot_uat` Grants (UAT Promotion)

DnA **owns** `pilot_uat` schemas and controls the promotion lifecycle. Teams do not have permanent write access — grants are **time-bound**, issued when a team is ready to promote, and revoked after validation completes.

**Baseline (permanent) grants:**

| Principal | `pilot_uat.team_a` | `pilot_uat.team_b` |
|---|---|---|
| `grp_team_a` | `USE SCHEMA`, `SELECT` | ❌ _no grant_ |
| `grp_team_b` | ❌ _no grant_ | `USE SCHEMA`, `SELECT` |
| `grp_team_a_business` | `USE SCHEMA`, `SELECT` _(validation only)_ | ❌ _no grant_ |
| `grp_team_b_business` | ❌ _no grant_ | `USE SCHEMA`, `SELECT` _(validation only)_ |
| `dna_admin` | `ALL PRIVILEGES` | `ALL PRIVILEGES` |

**Time-bound grants (during promotion window):**

| Principal | Grants added | Revoked after |
|---|---|---|
| `grp_team_a` | `CREATE TABLE`, `CREATE VIEW`, `CREATE MODEL`, `MODIFY` | Promotion + validation complete |
| `grp_team_b` | `CREATE TABLE`, `CREATE VIEW`, `CREATE MODEL`, `MODIFY` | Promotion + validation complete |

**Promotion workflow:**
1. Team requests promotion via SNOW ticket
2. DnA grants temporary write access to `pilot_uat.<team>` schema
3. Team promotes assets (CTAS / DEEP CLONE from `pilot_<team>` → `pilot_uat.<team>`)
4. Business users validate using their `SELECT` access
5. DnA revokes write grants, transfers object ownership to `dna_admin`

> Teams can **see schema names** in `pilot_uat` (they have `USE CATALOG`) but **cannot access** other teams' schemas (no `USE SCHEMA` cross-grant). MVs should not be promoted — promote base tables only. Model promotion follows a CI/CD pipeline (dev → UAT → prod).

> [!warning] Multi-Project Promotion Challenge: Many-to-One Schema Mapping
>
> In dev, each team has **multiple schemas** (one per project). In `pilot_uat`, each team has a **single schema**. This creates a many-to-one mapping during promotion:
>
> ```
> DEV (multi-schema)                    UAT (single schema)
> pilot_team_a/                         pilot_uat/
>   ├── project_x/  ──────────┐          └── team_a/  ← everything lands here
>   │     ├── customer_segments │               ├── ??? customer_segments
>   │     └── model_output      ├──promote──►   ├── ??? model_output
>   └── project_y/  ────────────┘               └── ??? invoice_summary
>         └── invoice_summary
> ```
>
> **Challenges with Option A (shared `pilot_uat`):**
>
> | Challenge | Detail |
> |---|---|
> | **Namespace collisions** | If `project_x` and `project_y` both have a `customer_segments` table, they collide. Requires naming conventions (e.g., `forecasting_customer_segments` vs `billing_customer_segments`) — convention over enforcement |
> | **Promotion scripts break** | Dev paths like `pilot_team_a.project_x.customer_segments` can't be reused — every notebook/query reference needs path rewriting (schema also changes, not just catalog) |
> | **Selective promotion is messy** | No schema boundary to separate projects. Cleanup after one project passes UAT means selectively dropping tables from a shared schema |
> | **Permission granularity loss** | In dev, access can be restricted per-project at schema level. In UAT, single schema means all team members see all promoted assets from all projects |
> | **Information leakage** | All teams have `USE CATALOG` on `pilot_uat`, so they can discover other teams' schema names — reveals what projects other teams are working on |
>
> **Option D solves this: per-team UAT catalogs**
>
> ```
> Option A (current):                   Option D (alternative):
> pilot_uat/                            pilot_uat_team_a/
>   └── team_a/                           ├── project_x/        ← mirrors dev exactly
>         ├── fx_customer_segments         │     ├── customer_segments
>         ├── fx_model_output              │     └── model_output
>         ├── fy_invoice_summary           └── project_y/
>         └── fy_anomaly_flags                   └── invoice_summary
> ```
>
> Option D preserves the same schema structure in dev and UAT — no naming conventions, no path rewriting, clean per-project promotion/cleanup, and full catalog-level isolation.
>
> **Recommendation:** Option A for initial rollout (~4-5 teams), but Option D becomes preferred if teams regularly run multiple projects through UAT concurrently. Design should be structured so migrating from `pilot_uat.<team>` to `pilot_uat_<team>` is low-effort. See [[Alinta Detailed Design - Catalog and Security#1.3 Note Potential Option D — Per-Domain UAT Catalogs]] for full analysis.

#### 5 — Security Job (Nightly Enforcement)

| Action | Detail |
|---|---|
| **Revoke creator `MANAGE`** | Any `MANAGE` grant on objects created by individual users → revoked. Ownership transferred to `grp_team_x` |
| **Revoke ad-hoc user grants** | Grants to individual users (not groups) outside the owning team → revoked |
| **Grant `SELECT` to team group** | Ensures all team members can read all objects within their schemas |
| **Validate volume ownership** | Volumes must be owned by `dna_admin` — any team ownership is corrected |
| **Audit cross-catalog grants** | Any `USE CATALOG` grant to a team on another team's catalog → revoked and flagged |

#### Summary: What Each Role Can Do

| Action | Team member | DnA admin | Security job |
|---|---|---|---|
| Create catalog | ❌ | ✅ (Terraform) | — |
| Create schema | ❌ | ✅ (Terraform) | — |
| Create table / view / model | ✅ (own schemas) | ✅ | — |
| Create volume | ❌ | ✅ (Terraform) | — |
| Read/write volume | ✅ (own schemas) | ✅ | — |
| Access other team's catalog | ❌ | ✅ | Revokes if found |
| Retain `MANAGE` on created objects | ❌ (revoked nightly) | ✅ | Enforces revocation |
| Promote to UAT | ✅ (time-bound write via SNOW request) | ✅ (grants + revokes access) | — |
| Read `prod` / `uat` | ✅ (`SELECT` only) | ✅ | — |
